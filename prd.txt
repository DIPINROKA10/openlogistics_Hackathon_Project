1. Executive Summary

To create the definitive AI evaluation environment for logistics and supply chain optimization, enabling researchers and developers to benchmark real-world operational decision-making capabilities.

OpenLogistics is a real-world logistics simulation environment built on the OpenEnv standard that evaluates AI agents' capabilities in multi-step planning, spatial reasoning, and constraint optimization. It transforms AI evaluation from text-based tasks to real-world decision-making simulation — testing whether an AI can plan, adapt, optimize, and execute like a real operations manager.

4.3 Data Flow
The interaction cycle follows a standard Gym-style reinforcement learning loop:
Agent sends POST /reset to initialize the environment for a specific task
Agent retrieves current state via GET /state
Agent submits an action via POST /step; environment validates, executes, calculates reward, checks termination
Cycle repeats until done=true (episode ends)
Agent calls GET /grade to retrieve the final normalized score

5.2 Environment Configuration

5.3 Graph Structure
The logistics network is modeled as a weighted undirected graph where nodes are warehouses and edges are routes with associated distances. Route availability can change dynamically in the Hard task.
Task 1 (Easy): 2-node simple network, no blocked routes
Task 2 (Medium): 3-node network, 1 pre-blocked route
Task 3 (Hard): 4-node hub-and-spoke network, 3 initially blocked + 2 dynamically blocked


6. Data Models

6.1 State Model
The StateModel is the canonical representation of the environment at any point in time, returned by /state, /reset, and /step.

StateModel {
    task_id:      str                    # Current task identifier
    time_step:    int                    # Current step (0 to max_steps)
    done:         bool                   # Episode termination flag
    warehouses:   List[Warehouse]        # All warehouse nodes
    trucks:       List[Truck]            # All truck assets
    orders:       List[Order]            # All pending/completed orders
    routes:       List[Route]            # All route edges with status
    metrics:      Metrics                # Cumulative performance stats
}

7. Task Design

7.1 Task Comparison Matrix

7.2 Task 1: Basic Delivery (Easy)

Grading formula: score = delivered_items / total_items. A 100% delivery scores 1.0; zero delivery scores 0.0. This task validates that the agent can perform the basic load → move → unload sequence correctly.

7.3 Task 2: Multi-Order Optimization (Medium)

Task 2 Grading Formula:
  score = 0.5 x delivery_rate + 0.3 x cost_efficiency + 0.2 x time_efficiency

  delivery_rate    = delivered_orders / total_orders
  cost_efficiency  = 1 - (total_cost / max_cost)        [max_cost = sum of distances x 2]
  time_efficiency  = 1 - (avg_delay / max_delay)        [max_delay = 10 steps]

7.4 Task 3: Crisis Management (Hard)

Dynamic Crisis Events

Task 3 Grading Formula:
  score = 0.4 x sla_success + 0.3 x adaptability + 0.3 x efficiency

  sla_success   = on_time_deliveries / total_orders
  adaptability  = routes_replanned / total_disruptions
  efficiency    = (total_delivered x 2) / (total_items + total_cost)

The grading system follows a multi-stage pipeline:
Step 1: Calculate raw metrics — deliveries completed, cost incurred, time adherence
Step 2: Apply task-specific component weights (see formulas in Section 7)
Step 3: Normalize each component to [0.0, 1.0]
Step 4: Compute weighted sum as final score
Step 5: Clamp to [0.0, 1.0] to handle edge cases

11. Baseline Agent (Inference)

11.1 Agent Architecture
The baseline agent is implemented in inference/agent.py and uses a Large Language Model (LLM) to decide actions at each step. It follows a standard perceive → reason → act loop:
Perceive: Call GET /state to retrieve the full environment state
Reason: Format state as a structured prompt and query the LLM for the next action
Act: POST the LLM's chosen action to /step
Repeat until done=True, then call GET /grade

11.2 Prompt Engineering
The system prompt provides the LLM with complete task context including warehouse inventory, truck positions, order list with deadlines, and the set of valid action types. The agent is instructed to return a single JSON action object at each step.

12. Deployment Strategy

12.1 Docker Configuration
The system is fully containerized using Docker and Docker Compose. A single docker-compose.yml orchestrates both the FastAPI backend and any supporting services.

The environment is deployable to Hugging Face Spaces using the Docker SDK, enabling public access for researchers and hackathon participants without local setup requirements.
